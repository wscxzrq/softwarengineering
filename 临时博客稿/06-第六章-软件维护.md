# 第六章 软件维护


## 软件维护的分类

软件交付使用后，为**改正错误**或**满足新需求**而修改软件的过程。

**引起软件维护的原因**：

1. 用户使用中发现测试阶段未察觉的错误
2. 软件运行环境变化（如操作系统 / 设备更新）
3. 用户产生新的使用需求

**软件维护分类**

1. **改正性维护**：识别并纠正错误、优化性能  <font color="#F56C6C">【改正错误】</font>
2. **适应性维护**：适配外部环境变化  <font color="#F56C6C">【适应环境】</font>
3. **完善性维护**：满足用户新功能 / 性能需求  <font color="#F56C6C">【完善功能】</font>
4. **预防性维护**：提升软件可维护性、可靠性   <font color="#F56C6C">【预防可维护性、可靠性】</font>

**交付之后改与扩，错漏、环境、新需求；改正适应完善防，四类维护保运行**

## 影响软件维护工作量的因素

1. **软件系统的规模**：系统越大、功能越复杂，维护工作量越高
2. **系统设计语言**：强功能编程语言可控制程序规模，降低维护成本
3. **系统年龄**：老系统的维护工作量远大于新系统
4. **数据库技术的应用**：简化数据管理与存储，减少报表类软件的维护工作量
5. **先进的软件开发技术**：面向对象、软件复用等技术，大幅减少维护工作量

**规模大、语言弱、系统老，维护工作量就高；数据库 + 先进技术，维护负担能减少**

## 软件维护的组织与申请

确立**正式 / 非正式维护机构**，保障维护工作有序开展。

**维护申请报告**：

1. 需修改变动的**性质**
2. 申请修改的**优先级**
3. 完成维护所需的**工作量**
4. 修改后的**预计状况**

**软件维护流程**：

1. 提交申请报告给**维护管理员**
2. 管理员转交报告给**系统监督员**（熟悉产品的技术人员）做评价
3. **修改负责人**根据评价确定修改方案
4. 维护人员实施修改，**配置管理员**把控修改范围，评审软件配置

**建机构，提申请（性、优、量、况）；管员转，监督员评，负责人定，配管员控**

## 程序的修改

1. **分析理解程序**：自顶向下梳理程序结构 + 数据结构

   1. **分析程序结构图**：分析程序接口，评估修改复杂度
   2. **数据跟踪**：跟踪过程内变量，理清数据流向。

2. **修改程序**：

   1. **设计程序的修改计划**：
      1. 全局分析模块、接口、数据库，制定整体修改方案；
      2. 分离待修改及受影响的模块 / 数据结构；
      3. 细化内部逻辑，明确新旧逻辑改动点；
      4. 提供用户回避措施，局部化问题，保障业务不中断。
   2. **修改程序**
      1. 正确编写修改代码，保持原有程序风格 / 格式，标注改动指令；
      2. 不删未确认无用的语句，不共用临时变量（自行新增）；
      3. 插入错误检测语句，记录修改细节，消除有害副作用。
   3. **重新验证程序**
      1. **静态确认**：至少 2 人参与，规避新错误；
      2. **计算机确认**：基于静态确认做修改后的确认测试；
      3. **维护后的验收**：维护主管部门检验后再交付。

   **先析结构与数据，再定计划改程序（保风格、不删改、插检测），最后双人静态验 + 机测 + 验收**

## 修改程序的副作用

1. **修改代码的副作用**：修改源代码时，不慎引入新的程序错误。
2. **数据的副作用**：调整数据结构后，导致软件设计逻辑与新数据结构不兼容、不匹配。
3. **文档的副作用**：程序修改后未同步更新相关文档，造成文档描述与软件实际功能脱节。

## 软件的可维护性

1. **定义**：**纠正软件错误 / 缺陷、满足新需求时修改软件的容易程度**。
2. 评价指标
   1. **可理解性**：核心是**读源码 / 文档易理解功能与结构**；关键特性：模块优、文档简，风格一致，代码清晰，**命名规范**
   2. **可靠性**：核心是**给定时间内按要求正确执行的概率**；核心度量：**平均失效间隔时间、平均修复时间、有效性**
   3. **可测试性**：核心是**易论证程序正确性**
   4. **可修改性**：核心是**程序易修改**
   5. **可移植性**：核心是**易转移到新计算环境**；关键特点：结构好、灵活，**不依赖特定硬件 / 系统**
   6. **效率**：核心是**完成功能且不浪费资源**；关键资源：内存、外存、通道、时间
   7. **可使用性**：核心是**程序方便、实用、易用**

**理可靠，测改移，效易用**

## 提高软件可维护性的方法

1. **建立明确的软件质量目标与优先级**：按程序用途区分质量特性重要性（如编译程序重效率，管理系统重易用 / 可修改），明确优先级以控成本、提质量。
2. **采用优质技术与工具**：
   1. **模块化**：功能独立，修改影响小；
   2. **结构化设计**：标准化模块结构与交互，优化程序结构。
3. **开展明确的质量保证审查**：
   1. **设检查点复审**，验证是否符合标准；
   2. **验收检查覆盖 4 大标准**：需求规范（可测试、分优先级）、设计（高内聚低耦合、可扩缩）、源代码（高级标准语言、结构化、文档化）、文档（说明输入输出、算法等）。
4. **选易维护语言 + 改进文档**：
   1. 优先高级语言（低级语言难维护），如 AI 应用选 Python 而非 C/C++；
   2. 优质文档需风格一致、易更新，程序自身也应具备文档性。


## 练习

1. 任何软件交付使用后都可能需要进行软件维护，下列关于引起软件维护的原因中，错误的是

   1. 软件投入运行时间过长 ✅
   2. 软件交付使用后发现了新的错误
   3. 软件使用一定时间后，用户提出了新的需求
   4. 软件的运行环境发生变化，需要进行软件的迁移

2. 下列选项中，因用户提出新的功能需求而进行的软件维护所属的类型是

   1. 完善性维护 ✅
   2. 适应性维护
   3. 预防性维护
   4. 改正性维护

3. 下列选项中，会导致软件维护工作量和难度变大的因素是

   1. 软件文档丰富、详细
   2. 采用高级程序设计语言
   3. 采用结构化软件开发技术
   4. 采用汇编语言编写基于网络的App ✅

4. 下列角色中，软件维护申请报告应该提交给谁

   1. 配置管理员
   2. 系统监督员
   3. 维护管理员 ✅
   4. 程序维护人员

5. 关于软件维护工作机构的组织，下列叙述中正确的是

   1. 软件维护机构的工作不需要用户参与
   2. 软件维护机构中任何人只能承担一种角色的工作
   3. 软件维护机构中的维护管理员也可以参与程序修改工作 ✅
   4. 必须建立正式的软件维护机构或部门以承担软件维护工作

6. 下列选项中，属于软件维护中修改程序所产生的副作用有

   1. 程序注释不详细
   2. 文档内容与程序不一致 ✅
   3. 程序的算法性能下降
   4. 软件的灵活性变差
