# 第七章 软件项目管理


## 软件项目管理的对象

软件项目管理的对象：人员（People）、产品（Product）、过程（Process）、项目（Project）

## 软件项目团队

1. **团队人员构成**

   1. **高级管理者（产品负责人）**：定义核心业务问题，把控项目方向
   2. **项目或技术管理者**：激励、组织、管理开发人员
   3. **开发团队成员**：具备产品开发所需专业技能
   4. **客户**：明确软件需求，含所有项目利益相关者
   5. **最终用户**：软件发布后直接交互的使用者

2. **团队的组织**

   **规划软件工程团队结构时应该考虑的7个项目因素**：

   1. 待解决问题的难度
   2. 程序规模（代码行 / 功能点衡量）
   3. 团队成员协作时长
   4. 问题模块划分的可行性
   5. 系统质量与可靠性要求
   6. 交付日期的严格程度
   7. 项目所需要的友好交流程度

3. **团队沟通**：

   建立正式 + 非正式交流机制，协调成员关系，保障信息同步

## 软件产品

1. **软件范围**：

   1. **核心要求**：项目初期需明确软件的**功能、性能范围及边界**
   2. **关键目标**：为项目经理**定量估算成本、有组织规划项目进展**提供依据

2. **问题分解**：

   又称 “问题划分 / 问题细化”，需覆盖两方面：

   1. 明确**必须交付的功能和内容**（产品输出物）
   2. 确定**所使用的软件过程模型**（开发方法论）

**产品先定范围（功、性、边），算本规划靠它；分解问题两要点，交付内容 + 过程模型**

## 软件过程模型

1. 核心要求：项目初期需结合项目特点选择合适过程模型（如瀑布模型、原型模型等）
2. 选择依据 3 大因素：
   1. 产品客户与开发团队人员
   2. 产品本身特性
   3. 软件团队项目工作环境

**记忆口诀：过程模型早选定，人、品、环境三因素定**

## 软件项目管理的工作内容

1. **启动软件项目**
   1. 确定项目的目标和范围
   2. 考虑可能的解决方案
   3. 确定合理、精确的成本估算
   4. 进行切实可行的任务分解
   5. 制定可管理的进度安排
2. **度量**：目的是把我软件工程过程的实际情况和它所产生的产品质量
3. **估算**：项目需要的人力、时间、成本进行估算
4. **风险分析**：识别项目潜在风险（如需求变更、技术难题、进度延误等），评估风险影响程度，制定应对与规避方案
5. **进度安排**：识别任务，估算每个任务工作量，分配资源，指定进度计划
6. **追踪和控制**：实时跟踪项目进度、成本、质量等关键指标，对比计划与实际执行情况；发现偏差时及时分析原因，采取调整措施（如重新分配资源、优化任务流程），确保项目按目标推进

**记忆口诀：项目启动定目标，度量估算风险找；进度安排明节点，追踪控制纠偏差**

## 软件项目管理的特点

1. **软件项目的特点**：软件产品无形、无物理属性，难以理解和驾驭。

2. **软件项目管理的困难**：

   1. 智力密集，可见性差
   2. 单件生产，难以复制
   3. 劳动密集，自动化程度低
   4. 程序编写繁琐，维护困难
   5. 软件工作渗透了人的因素：与人的水平、情绪、心理素质相关

3. **造成软件项目失败的原因**：

   大多与管理工作有关

4. **软件项目管理的主要职能**：

   1. **制定计划**：明确任务、要求、资源、人力及进度；
   2. **建立组织**：搭建分工明确的责任制机构；
   3. **配备人员**：任用各层次技术与管理人员；
   4. **指导工作**：激励团队完成分配任务；
   5. **检验工作进展**：对照计划 / 标准监督检查实施情况。

**记忆口诀：软件无形难驾驭，管理五难要记熟（智密可见差、单件难复制、劳密自动化低、编繁维护难、人因影响大）；失败多因管理误，五职（计划、组织、配人、指导、检验）保障项目路**

## 软件度量

**关注范围**：软件项目管理中，重点关注 **生产率度量** 与 **质量度量**—— 基于投入工作量，度量软件开发活动过程及成果质量。

**度量的核心目的**：

1. 明确软件产品质量水平；
2. 厘清软件开发生产率；
3. 验证新软件工程方法 / 工具在生产率、质量上的效益。

**两类核心度量分类**

1. 按度量维度分：生产率度量、质量度量、技术度量；
2. 按度量对象分：面向规模的度量、面向功能的度量、面向人的度量。

**度量核心产与质，三项目的要记实（明质量、清效率、验方法）；分类维度有两类，维度（产、质、技）对象（规模、功能、人）要分清**

## 面向规模和功能的度量

**面向规模的度量**：软件开发过程的**直接度量**，聚焦开发过程中的量化指标（如代码行数）

**面向功能的度量**：软件及开发过程的间接度量，核心关注程序的功能性与实用性，包含 5 个核心统计项：

1. 用户输入数
2. 用户输出数
3. 用户查询数
4. 文件数
5. 外部接口数

**规模直接量过程，功能间接看实用；输入输出加查询，文件接口五要素**

## 软件质量的度量

1. **正确性**：软件能精准执行需求规定的功能。
2. **可维护性**：软件维护所需工作量远高于其他软件工程活动。
3. **完整性**：衡量系统抵御安全攻击的能力，在网络安全时代愈发重要。
4. **可使用性**：核心是**用户友好性**，体现软件的易用程度。

**正确执行功能，维护耗工最多，完整抵御攻击，可用友好便捷**

## 风险评估

**风险驾驭核心流程**

风险识别 → 风险估计 → 风险评价 → 风险驾驭与监控

**各环节核心内容**

1. **风险识别**：聚焦 5 大商业风险
   1. **市场风险**：软件优秀但缺市场、缺用户
   2. **战略风险**：软件不契合产品整体战略
   3. **销售风险**：销售部门无有效推销方案
   4. **支持风险**：因课题 / 人员变动失去上级支持
   5. **预算风险**：资金相关风险
2. **风险估计**：从 3 个维度评估
   1. **风险表现**：风险发生时可能出现的具体问题
   2. **风险范围**：结合严重性与分布，衡量对项目、用户的影响程度
   3. **风险时间**：风险影响的起始时间及持续时长
3. **风险评价**：核心技术是定义风险参照水准（成本、进度、性能）
4. **风险驾驭与监控**
   1. **风险驾驭**：用技术方法避开或转移风险
   2. **风险监控**：对项目进行追踪的活动

**风险驾驭四步走，识别估计评价后；驾驭监控收尾留；**

**识别五险（市、战、销、支、预）要记牢，估计三维（表现、范围、时间）不可少；**

**评价基准（本、进、性）来参照，驾驭转移或避开，监控追踪不松懈**

## 成本评估

1. **软件开发成本估算方法**

   1. **自顶向下的估算方法**：

      从项目整体类推，依据过往完成项目的总成本 / 工作量，推算新软件的总成本 / 总工作量。

   2. **自底向上的估算方法**：

      将软件细分至子任务，明确每个子任务的开发工作量，累加得到总工作量。

   3. **差别估算方法**：

      与过往项目类比，区分相似与不同部分；相似部分按实际量估算，不同部分单独测算。

2. **软件开发成本估算的经验模型**

   1. **IBM模型**：

      静态单变量模型，基于源代码行数等特性估算资源需求量。

   2. **Putnam模型**：

      动态多变量模型。

   3. **COCOMO模型**：

      精确易用的静态单变量成本估算方法。

**估算方法三兄弟，顶下类推底上累，差别类比分异同；**

**经验模型有三种，IBM 静态单变量，Putnam 动态多变量，COCOMO 精准又好用**

## 进度安排的方法

1. 甘特图：表明了各任务的计划进度和当前进度

2. PERT和CPM

   1. PERT：计划评审技术
   2. CPM：关键路径法（PERT）

   PERT和CPM提供了以下定量工具

   1. 确定关键路径（CPM）：从开始到结束要花时间最长的那一串任务
   2. 应用统计模型（PERT）
   3. 计算边界时间

   1. 软件的质量和成本
   2. 软件产品和设计方案
   3. 软件产品和过程
   4. 软件的功能和性能 ✅

2. 下列因素中，影响开发团队选择软件过程模型的是

   1. 产品本身的特性 ✅
   2. 软件开发所使用的工具
   3. 产品的应用领域
   4. 软件的设计方案

3. 下列关于软件项目管理的叙述中，正确的是

   1. 软件项目团队中的技术人员越多，软件开发的效率越高
   2. 软件项目管理的对象仅包括软件开发过程和软件产品质量
   3. 软件项目管理需要对软件的风险进行监控 ✅
   4. 软件项目管理的重点是保证软件的开发进度

   1. IBM 模型是静态单变量模型 ✅

   3. IBM 模型是动态单变量模型
